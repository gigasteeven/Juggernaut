# ПРОМПТ ДЛЯ ИИ: мод Geometry Dash на Geode (двойной рендер + layout XDBot + Spout)

## КОНТЕКСТ И ВЕРСИИ (не менять!)
- Целевая игра: Geometry Dash 2.2081 (Windows).
- Фреймворк: Geode SDK 5.7.1. Используй только синтаксис и API, актуальные для Geode 5.7.1. Не используй устаревшие конструкции из старых версий Geode.
- Язык: C++ (как принято в Geode).
- В корневой папке проекта лежит исходный код мода XDBot (форк XDBotFork). Ты ДОЛЖНА опираться на его реализацию layout mode (файлы src/hacks/layout_mode.cpp и src/hacks/layout_mode.hpp). Внимательно изучи их перед написанием кода.

## ЧТО НУЖНО СДЕЛАТЬ (суть мода)
Сделать так, чтобы:
- На мониторе игрока уровень отображался в layout-режиме XDBot (голубой фон, только белые силуэты solid/hazard-объектов, декорации скрыты, без glow/pulse/шейдеров, без мигания).
- В OBS при записи тот же уровень отображался полностью, со всеми декорациями (как обычный уровень).
- Индикаторы (FPS и прочие оверлеи) должны быть видны в OBS-картинке.
- Не должно быть лагов и вылетов. Целевое железо: RTX 3090 (можно нагружать).
- Мод действует только на уровне (PlayLayer). В меню и редакторе всё работает штатно, без вмешательства.

## АРХИТЕКТУРА (обязательная схема — Вариант B: два PlayLayer в одном процессе)
Внутри одного процесса игры существуют ДВА игровых слоя одного и того же уровня:

1. PRIMARY (основной) — то, что игрок видит на мониторе.
   - Это обычный PlayLayer с полностью включённым layout-режимом XDBot (через его существующую механику модификации строки уровня).
   - Здесь работают все функции XDBot.
   - Игрок управляет именно этим слоем.
   - **Синглтон PlayLayer::get() игры должен указывать на PRIMARY.**
   - Звук (клики, музыка) итолько отсюдада**.
SHADOW (теневой)й)** — то, что уходит в OBS.
   - Второй PlayLayer того же GJGameLevel, нОРИГИНАЛЬНОЙ (немодифицированной) строкой уровняня** — полные декорации.
   - Находивне видимой сценыны**, не является активным синглтоном.
   - Рендеритсoffscreen render texturere**, поверх неё рисуются индикаторы, и результат отдаётсSpout2t2** → OBS получает его через источSpout2 Capturere** (НЕ Game Capture).
 Не играет звук.к.**

## ВАЖНО ПРО XDBot
- Layout XDBot работает чемодификацию строки уровня перед загрузкойой** (в PlayLayer::init) + обход LevelTools::verifyLevelIntegrity + правки в PlayLayer::addObject (отвязка от цветовых каналов, m_hasNoGlow = true, вырезание визуальных триггеров, удаление деко по спискам ID).- Layout должен применяться ТОЛЬКО к PRIMARY. Введи флаг «это shadow», чтобы XDBot:
  - НЕ модифицировал строку уровня для SHADOW (он должен быть полным);
  - НЕ применял bypass integrity к SHADOW (он на оригинальной строке, грузится штатно);
  - НЕ применял к SHADOW отвязку цветов/glow/вырезание триггеров.
- Не переписывай layout XDBot заново — используй его как есть на PRIMARY.

## СИНХРОНИЗАЦИЯ (критично! источник плавающих рассинхронов)
GD 2.2 считает физику фиксированным шагом (~240 тиков/с), отвязанным от FPS. Каждый кадр движок делает N фиксированных шагов в зависимости от накопленного dt.

ПРАВИЛА (нарушение = плавающий рассинхрон вида «то 2, то 10 кадров»):
- Единый источник времени. SHADOW НЕ должен иметь собственный аккумулятор dt и собственный schedule update.
- Каждый кадр SHADOW получает ровно тот же dt и ровно то же число фиксированных шагов, что и PRIMARY, и тот же ввод.
- SHADOW шагает строго в лок-степе с PRIMARY, разница ровно 0 кадров (НЕ «на 1 кадр позади»).
- В начале уровня выставить одинаковый random seed обоим слоям.
- Сначала синхронно прошагать оба слоя (логика), и только ПОТОМ рисовать. Рендер shadow в текстуру не должен влиять на его обновление.
- Физика игрока зависит только от solid/hazard (они одинаковы в обоих слоях), декорации на физику не влияют → при соблюдении правил рассинхрон детерминированно = 0.

ПРО ЗВУК КЛИКА: задержка «клик → звук» (например 4 кадра) — это асинхронная латентность аудиодвижка FMOD, она есть и в ванили, и к рассинхрону двух слоёв отношения не имеет. Звук берётся только из PRIMARY, поэтому двойного/расфазированного звука быть не должно.ОТЛАДКА СИНХРОНА: добавь debug-проверку — каждый кадр сравнивай позицию игрока (x/y) в PRIMARY и SHADOW; если разница хоть раз ≠ 0, логируй — значит тайминг разъехался.

## РЕНДЕР
- Создать offscreen render texture под выходное разрешение.
- Рендерить SHADOW (полный уровень) в эту текстуру (его visit, НЕ в экранный буфер).
- Поверх текстуры рисовать индикаторы (FPS и т.п.) — чтобы они попали в OBS.
- Отдавать текстуру в Spout2 как источник.
- PRIMARY (layout) рисовать на экран как обычно.

## ЖИЗНЕННЫЙ ЦИКЛ
- SHADOW создаётся только при входе в уровень и корректно уничтожается при выходе/рестарте (вместе с освобождением render texture и гашением Spout-источника, возврат синглтона на PRIMARY).
- В меню и редакторе SHADOW НЕ создаётся — всё штатно.
- Пауза — оба слоя на паузу одновременно.
- Рестарт/смерть — синхронно для обоих.

## ПОРЯДОК РЕАЛИЗАЦИИ (делай строго по шагам, не прыгай)
1. Научиться стабильно создавать второй PlayLayer того же уровня offscreen и уничтожать его при выходе — БЕЗ layout и БЕЗ Spout. Это сердце проекта. Если тут краши синглтона — стоп, решаем это в первую очередь.
2. Прикрутить зеркалирование инпутов + единый тайминг; убедиться через debug-проверку, что игрок в обоих слоях идёт кадр-в-кадр (разница 0).
3. Добавить рендер SHADOW в текстуру и временно вывести её на экран для визуальной сверки совпадения.
4. Включить layout XDBot только на PRIMARY через флаг/gate.
5. Навесить индикаторы на shadow-проход.
6. Подключить Spout2 вместо временного вывода, настроить источник в OBS.
7. Отполировать жизненный цикл (меню, редактор, пауза, рестарт, аудио только из PRIMARY).

## ГЛАВНЫЙ РИСК
GD не рассчитан на два живых PlayLayer. Синглтон PlayLayer::get() должен указывать на PRIMARY; SHADOW должен быть «отвязан» и не приниматься движком/XDBot за активный слой. Если шаг 1 даёт неустранимые краши — сообщи об этом явно, не маскируй костылями.
